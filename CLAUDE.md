## Project Overview

InfraGraph is an open-source cloud dependency mapper. Users upload Terraform plan/state JSON files; the backend parses them into a normalized resource graph stored in Postgres, then exposes APIs for graph exploration, blast radius analysis, and drift/issue detection.

## Commands

```bash
# First-time setup — creates backend/.venv and installs all deps
make install

# Start Postgres + API (Docker required, runs detached)
make dev              # starts Postgres + API on :8000 (hot-reload)
make down             # stop everything

# Run all backend tests (requires Postgres running)
make test

# Run a single test file
make test-one T=tests/test_plan_parser.py

# Lint and format
make lint
make format

# Database migrations
make migrate          # run pending migrations
make migrate-new MSG="add foo column"   # create new autogenerated migration
```

All `make` targets use the venv at `backend/.venv` automatically — no need to activate it manually.

## Architecture

**Backend** (`backend/`): Python 3.11+ / FastAPI with async SQLAlchemy 2.0 + asyncpg.

- `app/main.py` — FastAPI app entry point, mounts all routers under `/api`
- `app/config.py` — Settings via pydantic-settings, env prefix `INFRAGRAPH_`
- `app/db/models.py` — SQLAlchemy ORM models (Resource, Relationship, IngestionRun, Finding)
- `app/db/session.py` — async engine + session factory; `get_session` is the FastAPI dependency
- `app/routes/` — FastAPI routers: `ingest`, `resources`, `graph`, `findings`
- `app/services/` — Business logic: ingestion orchestration, graph traversal, blast radius, findings engine
- `app/adapters/terraform/` — Parsers for Terraform plan and state JSON; reference resolver for cross-resource relationships
- `app/schemas/` — Pydantic request/response models

**Data flow:** Terraform JSON → adapter/parser → normalized resources + inferred relationships → DB → graph/findings services → API responses

**Frontend** (`frontend/`): Next.js 14+ App Router with React Flow for graph visualization (Phase 2).

**CLI** (`cli/`): Typer-based CLI for ingest, serve, export, blast-radius commands (Phase 3).

## Data Model

Four core tables in Postgres:
- **resources** — normalized cloud resources with JSONB attributes/tags, indexed by `(external_id, resource_type)`
- **relationships** — directed edges between resources with type enum and confidence score
- **ingestion_runs** — tracks each parse/upload with status and counts
- **findings** — detected issues (orphan, drift, exposure, circular_dep, critical_node) linked to resources

Enums: `SourceType`, `RelationshipType`, `FindingType`, `Severity`, `IngestionStatus` — all defined as `StrEnum` in `app/db/models.py`.

## Key Conventions

- All DB operations are async (asyncpg + SQLAlchemy async sessions)
- Alembic is configured for async in `alembic/env.py` (adds `backend/` to sys.path)
- Tests use pytest-asyncio with `asyncio_mode = "auto"` — no need for `@pytest.mark.asyncio` on every test
- Test fixtures create/drop all tables per test via the `db_session` fixture in `conftest.py` (request it explicitly in DB-dependent tests)
- FastAPI dependency override pattern: tests override `get_session` to inject the test session
- Linting: ruff with rules `E, F, I, N, W, UP`, line length 100
- Relationship inference from Terraform attributes maps field names (e.g., `subnet_id`, `vpc_id`, `security_groups`) to typed edges between resources
